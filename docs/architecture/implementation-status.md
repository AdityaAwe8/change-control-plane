# Implementation Status

This document is intentionally reality-based. A subsystem is not considered complete because a directory, type, route, page, or document exists. The status below reflects actual code paths inspected in the repository.

Status legend:

- `verified`: implemented and proven by meaningful tests or executable flows
- `implemented_weakly_verified`: implemented, but with shallow tests or incomplete operational proof
- `partial`: important parts exist, but key runtime wiring or supporting surfaces are missing
- `stubbed`: mostly placeholder or cosmetic
- `missing`: not materially implemented
- `misleading`: present in docs or structure, but not actually functional

## Highest-Priority Gaps

1. Kubernetes and Prometheus are now local-reference proven for one advisory pilot path and have a stricter external proof runner plus a unified `make release-readiness` ship gate, but they are still not honestly customer-environment or production-environment proven until operators execute that runner against real environments and preserve the evidence bundle.
2. The reference pilot SCM path uses a local GitLab fixture. Hosted-like GitHub and GitLab harness proof is stronger now, the repository now has reusable saved-report validation for both pilot and live proof artifacts, and `make release-readiness` now classifies those artifacts explicitly, but real hosted GitHub and GitLab evidence still remains weaker than the local-cluster/local-metrics path until operators capture and preserve those runs.
3. Enterprise auth is now real as an OIDC foundation with HttpOnly browser sessions, server-side expiry/revocation, admin-scoped browser-session diagnostics/revocation, and safer callback handling, but SCIM, SAML, richer role mapping, and broader enterprise IAM breadth are still missing.
4. The policy subsystem is now materially real, but it remains deterministic and intentionally narrow: only `risk_assessment` and `rollout_plan` are policy-aware, policy results persist in the existing `policy_decisions` records rather than a new governance history engine, and there is still no broader external-policy compatibility layer.
5. Multi-instance integrations are now real, but overlapping repository attribution and deeper cross-instance topology reconciliation remain limited across both GitHub and GitLab.
6. Durable outbox eventing now has reclaim, dead-letter, operator-driven retry/requeue tooling for `error` and `dead_letter`, and stronger single-process runtime proof around stale claims and recovery races, but processed-event replay and broader multi-process operational proof remain incomplete.
7. Discovery is deeper than before: GitHub/GitLab sync now records deterministic CODEOWNERS ownership evidence, repository/discovered-resource mappings now carry provenance markers, and mapped services can infer owner-team edges into the graph. Deeper dependency inference, identity-backed ownership sync, broader workload discovery, and richer many-to-many SCM provenance still remain shallow.
8. Route-local truth now covers the authenticated SPA and page-owned state is the only route-data source, but the shell still coordinates those page models through one shared container and selector layer. Thinner routes intentionally remain low-fan-out route-local multi-request surfaces rather than gaining backend page-state bundles.
