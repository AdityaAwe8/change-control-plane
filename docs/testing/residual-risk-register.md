# Residual Risk Register

This register captures the highest-value risks that remain after the current verification pass.

| Risk | Current State | Why It Is Not Fully Closed | Risk Level | What Closes It |
| --- | --- | --- | --- | --- |
| Kubernetes provider is not customer-cluster proven | local-cluster proven plus external runner, preflight checklist, and report validation available | the reference pilot now proves discovery and observation against a local `k3s` cluster, and the repository now includes a reusable `live-proof-verify` runner, a secret-safe `make proof-live-preflight` checklist, and saved-report validation for customer-like environments, but no checked-in proof exists yet from a hosted or customer cluster with real auth, routing, and operational constraints | high | execute the external proof track against a real customer-like cluster and preserve the evidence |
| Prometheus provider is not hosted telemetry proven | local-metrics proven plus external runner, preflight checklist, and report validation available | the reference pilot now proves query windows and persisted signal snapshots against a local Prometheus deployment, and the repository now includes a reusable `live-proof-verify` runner, a secret-safe `make proof-live-preflight` checklist, and saved-report validation for customer-like environments, but no checked-in proof exists yet from a hosted or customer telemetry backend | medium | execute the external proof track against a real hosted/customer Prometheus environment and preserve the evidence |
| Reference pilot SCM proof is still fixture-based | local-reference proven, harness-proven, external runner available, and operator checklist available | the pilot proves GitLab discovery, webhook registration, and MR ingest against a local fixture, realistic fake upstreams now prove GitHub App install-style webhook registration/repair plus GitLab hosted-group repair, and the repository now includes a reusable external proof runner, a secret-safe preflight checklist with exact callback/webhook route expectations, and saved-report validation, but no real hosted GitHub or GitLab environment is exercised end to end from inspected sources | medium | execute the external proof track against real hosted SCM and preserve the report/evidence |
| GitHub App onboarding is not yet production-proven | harness-proven and partially live | install-style onboarding, signed state handling, installation-token exchange, disabled pre-install registration state, and post-install webhook registration/repair are tested against realistic fake upstreams, but not a real enterprise GitHub App deployment | medium | real GitHub App install validation and production hosted proof |
| GitLab onboarding is not yet enterprise-grade | repo-proven and harness-proven | token-based GitLab onboarding, discovery, webhook ingest, automatic webhook registration, and hosted-like group-webhook repair are now real, but there is still no OAuth flow or GitLab App-style onboarding | medium | real GitLab OAuth/App onboarding and hosted GitLab validation |
| Scheduled sync is not yet strongly proven under multi-worker contention | partially improved | duplicate-claim prevention is now tested in-repo, but not under true multi-process crash recovery or broader operational scale | medium | multi-worker lease contention tests and crash-recovery verification |
| Discovery is still shallow beyond runtime resource inventory | partial | repositories, workloads, signal targets, shared SCM repository attribution, deterministic CODEOWNERS ownership evidence, and mapped-service owner inference are now persisted, but richer dependency inference and identity-backed ownership sync are still missing | medium | deeper topology inference, identity-backed ownership sync, and broader workload provenance |
| Cross-instance repository attribution remains imperfect | partial | overlapping GitHub scopes now preserve a primary source integration plus provenance hints rather than a full many-to-many source model | medium | explicit cross-instance repo provenance model and conflict-resolution rules |
| Durable eventing is not yet a full replayable operations plane | partial | important events now persist in an outbox, reclaim stale claims, and dead-letter bounded failures, but replay tooling and external bus integration are still missing | medium | add replay/dead-letter flows and stronger multi-process operational proof |
| OpenAPI verification is still not a full schema/runtime diff | route, fragment, and near-exhaustive success-path runtime verified | contract tests now assert every registered HTTP route has an OpenAPI path/method entry and validate real serialized runtime payloads for nearly all registered success-path JSON surfaces across auth/system, catalog/governance, rollout/runtime, integrations/discovery, enterprise admin/public auth, and machine-auth lifecycle, but they still do not validate every alternate status code, redirect branch, or failure envelope exhaustively against the spec automatically | low | generated schema/runtime validation against every live handler and major error status |
| Browser enterprise auth is still pilot-grade | partial | OIDC sign-in and provider attribution are real, redirect/session handling now lands in persisted HttpOnly browser sessions with server-side expiry/revocation, org admins can now inspect and revoke active browser sessions, and expired sessions send the SPA back to sign-in with explicit feedback, but broader enterprise IAM proof, richer role mapping, and SCIM/SAML are still limited | medium | stronger enterprise auth proof, richer enterprise role/session administration, and SCIM/SAML expansion |
| CLI surface coverage is incomplete | partial | status, discovery, enterprise browser-session administration, integration schedule paths, machine-auth lifecycle commands, and team lifecycle commands are now tested, but most command groups remain only manually or indirectly exercised | low | add command-group tests for rollout, rollback-policy, and broader org/service/environment flows |
| CLI operator messaging is still JSON-first | partial | advisory semantics and provider evidence are clearer in the payloads, but not yet rendered as dedicated human-oriented CLI prose | low | add richer human-readable CLI summaries alongside JSON output or behind a flag |
